# 测试用例生成器
## 快速参考
### 输出格式
只输出 markdown 格式，支持三种输出模板：**结构化需求文档、标准测试用例表、用例评审报告**。

### 用例类型筛选
| 筛选词 | 效果 |
|-------|------|
| "只要正向"、"正向用例" | 只生成正向业务流程测试用例 |
| "不要异常"、"不要反向" | 排除异常、反向类测试用例 |
| "只要功能" | 仅生成功能测试用例，不输出边界/并发/网络等非功能用例 |

### 输入类型识别
| 类型 | 支持格式 |
|------|---------|
| 原始需求 | 纯文本自然语言业务需求描述 |
| PRD文档 | .docx, .md, .txt, .pdf |
| 设计图 | .png, .jpg（需配合文字业务描述） |
| 在线文档 | 任意可访问URL在线文档 |

## 核心功能
### 1. 原始需求收集与智能解析
自动接收零散自然语言原始需求、各类PRD文档，完成：
- 自动识别业务领域、业务模块、角色参与者
- 提取核心操作、业务规则、输入约束、权限限制
- 梳理隐含业务场景、字段属性、限制条件
- 输出结构化需求清单，解决原始需求散乱、信息缺失问题

### 2. 需求文档标准化整理
基于解析后的需求，自动完成整理与补全：
- 结构化规整：按模块、功能、业务规则、前置约束分层排版
- 智能场景补全：自动挖掘边界条件、异常场景、行业通用规则
- 逻辑校验梳理：理顺主流程、替代异常流程、后置结果
- 输出可直接用于评审的标准需求文档Markdown

### 3. 测试覆盖维度
| 类型 | 覆盖内容 |
|------|---------|
| 正向测试 | 核心业务主流程、正常场景验证 |
| 反向测试 | 非法输入、错误操作、异常业务分支处理 |
| 边界值测试 | 输入数值边界、数量上限、状态临界条件 |
| 等价类划分 | 有效/无效测试数据分组归类 |
| 场景测试 | 跨模块联动、完整业务链路场景 |
| 权限测试 | 未登录访问、越权操作、多角色权限校验 |
| 网络异常测试 | 断网、弱网、接口超时、请求失败处理 |
| 并发测试 | 重复提交、多端并发、数据竞争、库存并发冲突 |

### 4. 用例标准字段规范
**标准必填字段**：
| 字段 | 必填 | 说明 |
|------|-----|------|
| 用例编号 | ✓ | TC_模块_序号，如 TC_CART_001 |
| 所属模块 | ✓ | 归属业务模块名称 |
| 用例标题 | ✓ | 精简描述测试场景与操作点 |
| 用例类型 | ✓ | 功能测试/边界测试/异常测试/权限测试/并发测试 |
| 优先级 | ✓ | P0/P1/P2 |
| 前置条件 | ✓ | 执行用例前系统、用户、业务需满足的条件 |
| 测试步骤 | ✓ | 逐条可落地操作步骤，描述清晰无歧义 |
| 预期结果 | ✓ | 每一步对应明确可验证结果，无模糊表述 |

### 5. 优先级定义
| 优先级 | 占比 | 定义 |
|-------|------|------|
| P0 | 10-25% | 核心功能、主干业务流程，必测不可缺失 |
| P1 | 30-60% | 重要功能、主流用户常用场景 |
| P2 | 10-25% | 边缘功能、特殊小众场景、兼容兜底场景 |

### 6. 用例评审确认辅助能力
- 智能排查遗漏场景：边界条件、异常分支、业务规则、非功能约束
- 用例合理性校验：步骤冗余、预期结果模糊、优先级不合理检测
- 输出评审报告：遗漏清单、问题用例、优化建议、优先级调整建议
- 支持人机迭代：按指令合并/拆分用例、补充异常流程、调整粒度、修正描述

## 全流程标准工作流程
完整落地 **收集原始需求 → 整理需求文档 → 编写用例 → 评审确认** 闭环：
1. **收集原始需求**：接收自然语言/PRD/图片/在线文档，自动解析提取需求要点
2. **整理需求文档**：结构化排版、补全边界与异常场景、生成标准可评审需求文档
3. **编写测试用例**：按标准字段+多维度测试方法，批量生成规范Markdown测试用例
4. **评审确认优化**：自动检查遗漏与问题、输出评审意见，支持迭代修改直至最终确认落地

## 使用最佳实践
1. **分步执行不一步到位**
按：提取需求清单 → 人工确认范围 → 生成详细用例 → 排查遗漏场景 → 输出最终版，每步可人工介入校准，避免方向偏差。

2. **追问迭代优于重写**
输出不符预期时，直接在当前结果上指令补充/修改，无需重新写prompt，效率更高。

3. **角色设定调动领域知识**
可指定资深产品/测试分析师角色，带入行业业务规则，减少业务常识遗漏。

4. **AI为辅助，人做最终决策**
AI负责结构化、格式规范、场景补全；**业务范围取舍、优先级判定、规则准确性、评审最终确认必须由人把控**。

5. **保留迭代版本**
建议留存每版需求文档、测试用例、评审记录，方便后续需求变更与问题回溯。

## 能力边界与人机分工
### AI擅长
- 自然语言需求快速结构化、格式统一标准化
- 通用边界/异常/权限场景自动补全，减少人工遗漏
- 批量生成规范测试用例，节省重复编写时间
- 自动做用例格式、逻辑完整性初审与评审建议

### AI不擅长
- 不了解项目独有定制业务规则，可能编造不存在场景
- 无法自主判定需求优先级、业务范围取舍
- 不清楚现有系统架构、接口约束、历史兼容逻辑
- 无法感知业务隐性诉求、组织与项目约束

### 必须人工负责
- 业务场景纳入/剔除决策、需求范围管理
- 用例优先级排序、模块划分、用例合并拆分
- 业务规则真实性校验、与实际系统对齐
- 需求评审、用例最终签字确认
