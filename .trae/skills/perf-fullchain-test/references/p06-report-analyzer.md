# P06 报告分析 — perf-report-analyzer

## 解决的问题

JMeter 报告 20 多列指标不知先看哪个，瓶颈定位靠猜，优化建议写不具体。

## 触发词

- 压测报告分析 / JTL 分析 / 性能瓶颈定位
- perf report analyze / jtl analyzer / 压测结果分析

## 输入

- JMeter 聚合报告原始数据 / .jtl 文件 / 服务器监控数据 / 应用日志

## 输出

1. 指标异常诊断（哪些异常、程度、优先级）
2. 瓶颈定位决策树（四象限定位法）
3. 根因分析（具体到组件层次）
4. 优化建议（每个瓶颈 2~3 个方向 + 预期收益）

---

## 指标异常诊断

### 关键指标优先级

| 优先级 | 指标 | 异常阈值 | 说明 |
|--------|------|---------|------|
| P0 | 错误率 | > 0.1% | 最严重，先排查 |
| P0 | TPS 未达目标 | < 目标值 80% | 系统吞吐能力不足 |
| P0 | P95 超 SLA | > SLA 阈值 | 用户体验劣化 |
| P1 | P99 偏高 | > P95 × 2 | 存在长尾请求 |
| P1 | 平均 RT 偏离基线 | > 基线 × 1.5 | 性能退化 |
| P2 | 响应时间波动大 | 标准差 > 均值 × 0.5 | 系统不稳定 |

### 诊断规则

| 现象 | 可能的根因 |
|------|-----------|
| TPS 低 + RT 正常 | 并发数不足 / 压测机瓶颈 |
| TPS 低 + RT 高 | 服务端瓶颈（DB/CPU/IO） |
| TPS 达到瓶颈后 RT 飙升 | 系统达到极限容量 |
| 错误率和 RT 同时飙升 | 系统雪崩 / 连接池耗尽 |
| 错误率高但 RT 正常 | 业务断言异常 / 参数化数据问题 |
| 响应时间周期性抖动 | GC 停顿 / 定时任务 / 慢查询 |

## 瓶颈定位决策树

```
1. 错误率 > 0.1%？
   ├─ Yes → 检查错误码
   │        ├─ 4xx → 参数/认证问题
   │        └─ 5xx → 服务端异常，查看应用日志
   │        └─ Socket/Timeout → 连接池/网络问题
   └─ No → 检查 TPS

2. TPS 未达目标？
   ├─ 并发数未达预期 → 检查 JMeter 线程组配置
   ├─ 并发数已达但 TPS 低
   │   ├─ RT 正常 → 压测机瓶颈（CPU > 85%）
   │   └─ RT 高 → 检查服务端资源
   └─ TPS 达到平台期 → 进入容量评估

3. RT 超过 SLA？
   ├─ 检查服务器 CPU：
   │   ├─ CPU > 85% → 计算密集型/代码效率/线程争用
   │   └─ CPU 正常 → 检查 IO
   ├─ 检查磁盘 IO：
   │   ├─ iowait > 20% → DB 慢查询/日志刷盘
   │   └─ iowait 正常 → 检查网络/锁
   ├─ 检查 DB 慢查询：
   │   ├─ 有慢查询 → 索引缺失/查询量大/未分页
   │   └─ 无慢查询 → 检查 Redis/外部调用
   └─ 检查锁争用：
       ├─ 数据库行锁 → 并发写冲突
       ├─ Redis 热 key → 单 key 访问量过大
       └─ 分布式锁 → 锁等待严重
```

## 根因分析示例

### 数据库层

| 现象 | 判断依据 | 验证方法 |
|------|---------|---------|
| 慢查询 | RT 高 + DB CPU 高 + 慢查询日志 | `EXPLAIN ANALYZE` 检查索引 |
| 连接池满 | 错误包含 `Cannot get connection` | `show processlist` 查看活跃连接 |
| 行锁 | 高并发写 + RT 抖动 | `show engine innodb status` |
| 死锁 | 错误包含 `Deadlock found` | 查看 DB 错误日志 |

### 缓存层

| 现象 | 判断依据 | 验证方法 |
|------|---------|---------|
| 缓存穿透 | RT 高 + DB QPS 飙升 | 检查 Redis 命中率 < 90% |
| 缓存击穿 | 单 Key 过期后 RT 突增 | 检查热点 Key 的 TTL |
| 缓存雪崩 | 大范围 RT 飙升 + 大面积 Key 同一时间过期 | 检查 TTL 是否随机化 |
| 热 Key | 单 Redis 节点 CPU 高 | `redis-cli --hotkeys` |

### JVM 层

| 现象 | 判断依据 | 验证方法 |
|------|---------|---------|
| Young GC 频繁 | RT 周期抖动 + GC 日志 | `jstat -gcutil PID` |
| Full GC | STW 导致 TPS 断崖 | `-Xloggc` 分析 GC 停顿时间 |
| 内存泄漏 | OOM 或 GC 后内存不降 | `jmap -heap` 堆转储分析 |

## 优化建议模板

### 数据库优化

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 添加覆盖索引 | RT 降低 30%~70% | 低（DDL） |
| 分库分表 | 容量扩展 2~4 倍 | 高（架构改造） |
| 读写分离 | 读 QPS 提升 2~3 倍 | 中（配置变更） |
| 连接池调大 | 支持更多并发 | 低（配置变更） |

### 缓存优化

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 增加本地缓存（Caffeine） | RT 降低 50%~90% | 中（代码改造） |
| 热 Key 拆分 | 单节点压力降低 | 低（代码改造） |
| TTL 随机化 | 防止缓存雪崩 | 低（配置变更） |
| 布隆过滤器 | 解决缓存穿透 | 中（代码改造） |

### 代码/架构优化

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 异步化非核心链路 | RT 降低 20%~40% | 中（代码改造） |
| 线程池调优 | 吞吐量提升 20%~50% | 低（配置变更） |
| 限流熔断 | 防止雪崩 | 中（框架集成） |
| 连接池复用 | 减少连接建立开销 | 低（配置变更） |

## 跨平台 Prompt 模板

```
你是一个性能测试报告分析专家。根据用户提供的压测结果数据（JTL/聚合报告/监控数据），输出专业的分析报告。

请按以下步骤执行：

### 第一步：基础数据提取
提取以下核心指标：总请求数、成功率、TPS(平均/峰值/当前)、响应时间(平均/P50/P90/P95/P99/最大)、错误类型和分布

### 第二步：指标异常诊断
按优先级判断：
- P0: 错误率>0.1%、TPS<目标80%、P95超SLA
- P1: P99>P95×2、平均RT>基线×1.5
- P2: 响应时间标准差>均值×0.5

### 第三步：瓶颈定位（使用决策树）
错误率>0.1%? → 检查错误码(4xx/5xx/Socket)
TPS未达标? → 并发不足/服务端瓶颈/压测机瓶颈
RT超SLA? → CPU>85%/iowait>20%/慢查询/锁争用/GC

### 第四步：时间分片趋势分析
按每30秒切片，分析TPS、RT、并发数随时间的变化趋势
特别关注：初始高并发毛刺、中间TPS断崖、后半段性能衰减

### 第五步：根因判断
- 数据库: 慢查询/连接池满/行锁/死锁
- 缓存: 穿透/击穿/雪崩/热Key
- JVM: Young GC频繁/Full GC/内存泄漏
- 代码: 线程阻塞/锁竞争/序列化开销

### 第六步：优化建议
每个瓶颈给出 2-3 个优化方向（含预期收益和落地难度）

### 第七步：综合结论
按以下维度评分（★1-5）：
功能正确性 | 响应性能 | 吞吐能力 | 稳定性 | 连接管理
并给出是否可投产的综合判断
```
