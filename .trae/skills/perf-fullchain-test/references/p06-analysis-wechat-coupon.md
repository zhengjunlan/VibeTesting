## 第五步：根因分析

### 根因 1：缓存冷启动（影响时段：0~30s）

| 项目 | 内容 |
|------|------|
| **现象** | 前 30s P95=806ms，随后降至 400~500ms |
| **根因** | Redis/本地缓存未预热，第一批请求直穿 DB |
| **典型特征** | 冷 RT 是温 RT 的 1.5~2 倍 |
| **验证方法** | 预热后压测对比 P95 是否降至 400ms 以下 |
| **置信度** | 高（符合缓存冷启动特征曲线） |

### 根因 2：连接池配置未对齐（影响时段：30~60s）

| 项目 | 内容 |
|------|------|
| **现象** | 并发 250 时 TPS 骤降至 28.7，4 条 Socket closed |
| **根因** | 服务端/Tomcat/DB 连接池空闲超时与客户端 Keep-Alive 不匹配，服务端主动关闭连接后客户端复用失败 |
| **典型特征** | 非 HTTP 错误码的 Socket closed，仅在高并发切换时出现 |
| **验证方法** | 核对 `server.tomcat.keep-alive-timeout` vs JMeter `timeout`；DB 连接池 `idleTimeout` |
| **置信度** | 高（并发 250 释放连接时出现典型 Socket closed） |

### 根因 3：压测数据耗尽 / 服务端限流（影响时段：240~505s）

| 项目 | 内容 |
|------|------|
| **现象** | 并发 160 不变，TPS 从 160 持续降至 37，RT 稳定在 261~310ms |
| **根因** | 最可能：CSV 参数化数据（用户ID/优惠券）被耗尽，新数据不满足业务条件，请求快速失败或空返回 |
| **备选根因** | 服务端触发限流（令牌桶/漏桶），或 Redis 缓存热 Key 资源耗尽 |
| **典型特征** | RT 不升高但 TPS 下降（排除服务端过载），符合"数据不够→请求无效→快速返回"模式 |
| **验证方法** | 检查 CSV 数据量是否 ≥ 并发×TPS×时间；检查服务端限流日志 |
| **置信度** | 中高 |

## 第六步：优化建议

### 问题 1：缓存冷启动

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 压测前执行数据预热脚本（预跑 1~2 分钟 20% 并发） | 初始 P95 从 806ms→450ms（↓44%） | 低（配置变更） |
| 应用层增加启动预热机制（`@PostConstruct` 加载热点数据） | 生产上线时初始请求无毛刺 | 中（代码改造） |

### 问题 2：连接池耗尽

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 对齐 Tomcat Keep-Alive Timeout（建议 60s）与客户端参数 | 消除 Socket closed 错误 | 低（配置变更） |
| 调大 HikariCP 连接池最大连接数（> 250） | 支持更高并发不等待 | 低（配置变更） |
| 增加 `spring.datasource.hikari.max-lifetime` < 服务端 idle-timeout | 避免服务端主动关闭 | 低（配置变更） |

### 问题 3：数据耗尽

| 优化方向 | 预期收益 | 落地难度 |
|----------|---------|---------|
| 压测前按公式计算所需数据量并提前构造 → `并发×TPS×时间×1.2` | 保证压测全程数据充足 | 低（流程规范） |
| 参数化 CSV 使用 Random CSV 或组合多张 CSV 轮询 | 延长数据使用时长 | 低（脚本改造） |
| 写接口使用 `__UUID()` 或时间戳生成唯一标识 | 避免重复数据导致断言失败 | 低（脚本改造） |
